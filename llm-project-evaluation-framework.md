# LLM 项目评估与应用框架文档

## 1. 文档目标

本文档用于指导团队使用 LLM 模型对项目进行评估，并在评估基础上完成应用框架设计与 MVP 落地规划。

适用场景包括：

- 基于业务文档、知识库或代码仓库构建智能问答系统
- 使用 LLM 提升现有业务流程自动化能力
- 规划 Prompt、RAG、Workflow 或 Agent 类应用
- 在项目启动前形成结构化评估报告和实施方案

---

## 2. 总体方法

建议按以下两条主线并行推进：

- 项目评估：判断项目是否适合引入 LLM，以及价值、成本和风险是否可控
- 应用框架设计：明确系统如何搭建、先做哪些能力、技术路线如何落地

整体实施建议分为 6 个阶段：

1. 明确业务目标与成功标准
2. 识别适合接入 LLM 的业务场景
3. 评估数据、技术、成本和风险
4. 确定应用类型与系统架构
5. 设计 MVP 范围与实施路线图
6. 建立评测、监控和持续迭代机制

---

## 3. 项目评估步骤

### 3.1 明确项目目标

在引入 LLM 之前，先回答以下核心问题：

- 要解决什么业务问题
- 目标用户是谁
- 当前流程中最耗时、最重复、最依赖人工判断的环节是什么
- 引入 LLM 后希望提升哪些指标
- 如何定义项目成功

建议输出内容：

- 项目背景说明
- 用户角色定义
- 当前流程问题清单
- 业务目标和成功标准

### 3.2 划定高价值场景

优先选择高价值、低风险、数据相对充足的场景进行验证。常见场景包括：

- 文档问答与知识检索
- 内容摘要与信息提取
- 代码理解与辅助分析
- 业务流程问答助手
- 工单分析与自动分类
- 报告生成与规范化输出

场景选择原则：

- 用户需求明确
- 输入输出边界清晰
- 可通过样例验证效果
- 容易形成闭环反馈
- 对错误容忍度可控

### 3.3 评估数据条件

LLM 项目是否可落地，关键取决于数据基础。需要确认：

- 是否已有业务文档、FAQ、知识库、历史记录或代码仓库
- 数据是否完整、可访问、可更新
- 数据是否存在敏感信息或权限限制
- 是否需要结构化清洗、分段、标注或索引

建议梳理的数据来源：

- 产品文档
- 业务规则文档
- API 文档
- 数据库表结构说明
- 日志与历史工单
- 代码仓库与技术文档
- 第三方资料与标准规范

建议输出内容：

- 数据源清单
- 数据可用性评估
- 敏感信息识别结果
- 知识更新机制建议

### 3.4 评估技术可行性

需要判断当前问题更适合以下哪种技术路径：

- 纯 Prompt：适合简单生成、改写、摘要、分类
- RAG：适合基于文档、知识库、代码仓库进行问答
- Workflow：适合固定流程、多步骤任务编排
- Agent + Tool Calling：适合需要调用外部系统执行任务
- 传统规则或传统模型：适合稳定、可枚举、解释性强的任务

判断标准：

- 是否依赖外部知识
- 是否需要调用工具或系统
- 是否需要多轮推理
- 是否对稳定性要求极高
- 是否可接受一定的生成误差

### 3.5 评估成本与收益

成本评估建议至少包含以下维度：

- 模型调用成本
- 向量检索和存储成本
- 开发与测试成本
- 运维和监控成本
- 知识更新维护成本

收益评估建议关注：

- 是否减少人工时间
- 是否提升任务完成率
- 是否缩短响应时间
- 是否提升用户满意度
- 是否带来流程自动化收益

### 3.6 评估风险与限制

LLM 项目常见风险包括：

- 幻觉和错误引用
- 敏感数据泄露
- 权限越权访问
- 输出不稳定
- 上下文不足导致误判
- 成本不可控
- 合规和审计要求无法满足

建议建立风险台账，至少记录：

- 风险描述
- 影响范围
- 触发条件
- 规避方案
- 应急处理策略

## 4. 判断适合的应用类型

在完成项目评估后，需要选择合适的应用模式。

### 4.1 Prompt App

适合场景：

- 文本生成
- 摘要
- 分类
- 改写
- 初步验证业务价值

特点：

- 开发最快
- 成本最低
- 易于搭建原型
- 对知识依赖较弱

### 4.2 RAG 应用

适合场景：

- 文档问答
- 知识库助手
- 项目资料检索
- 代码仓库分析

特点：

- 能利用私有知识
- 相比纯 Prompt 更可控
- 需要额外建设知识处理链路

### 4.3 Workflow 应用

适合场景：

- 固定流程自动化
- 多步骤审批与处理
- 标准化任务编排

特点：

- 稳定性较高
- 便于控制流程
- 适合企业业务落地

### 4.4 Agent 应用

适合场景：

- 需要调用多个外部工具
- 需要自动规划任务步骤
- 需要复杂决策和执行能力

特点：

- 能力上限高
- 实现复杂度高
- 稳定性与可控性要求更高

建议优先级：

- 第一阶段优先做 Prompt 或 RAG
- 第二阶段引入 Workflow
- 第三阶段在必要时引入 Agent

---

## 5. 应用框架设计

一个可落地的 LLM 应用框架，建议至少包含以下 7 层：

### 5.1 接入层

负责用户入口与系统接入：

- Web 前端
- 管理后台
- API 接口
- 企业微信/飞书/钉钉等消息入口

### 5.2 应用层

负责具体业务能力封装：

- 智能问答
- 文档分析
- 报告生成
- 数据提取
- 任务执行

### 5.3 编排层

负责请求处理和能力组织：

- Prompt 模板管理
- 对话上下文管理
- Workflow 编排
- Agent 控制逻辑
- 模型路由策略

### 5.4 模型层

负责模型接入与调用：

- 通用大模型
- 专用领域模型
- 本地部署模型
- 多模型切换与降级

### 5.5 知识层

负责知识处理与检索：

- 文档解析
- 文本切片
- Embedding 生成
- 向量索引
- 混合检索
- 重排与引用返回

### 5.6 工具层

负责外部能力连接：

- 数据库查询
- 文件系统
- 搜索服务
- 内部 API
- 第三方系统
- 安全审计接口

### 5.7 治理层

负责稳定性与安全治理：

- 日志记录
- 监控告警
- Prompt 版本管理
- 评测体系
- 成本控制
- 权限审计
- 缓存与限流

---

## 6. 参考架构

以下是一个常见的 LLM 应用逻辑结构：

```text
前端界面 / API 接入
    -> 应用服务层
        -> LLM 编排服务
            -> Prompt 管理
            -> 对话上下文管理
            -> RAG 检索服务
            -> Tool Calling
            -> 结果后处理
        -> 数据服务层
            -> PostgreSQL
            -> Redis
            -> Vector DB
            -> 文件存储
        -> 治理与观测
            -> 调用日志
            -> 反馈采集
            -> 评测系统
            -> 监控与告警
```

## 7. 技术选型建议

如果目标是快速搭建 MVP，可参考如下技术组合：

### 7.1 后端

- FastAPI
- Node.js + NestJS

### 7.2 前端

- React
- Next.js

### 7.3 编排与开发框架

- LangChain
- LlamaIndex
- Dify
- Flowise
- 自定义轻量编排层

### 7.4 数据与存储

- PostgreSQL
- Redis
- pgvector
- Qdrant
- Milvus
- 对象存储或本地文件存储

### 7.5 评测与观测

- Langfuse
- OpenTelemetry
- 自建日志平台
- 自定义反馈标注系统

技术选型原则：

- 优先满足当前 MVP，而不是一次性追求平台化
- 优先选择团队熟悉的技术栈
- 业务逻辑应尽量与底层 LLM 框架解耦
- Prompt、工具定义、知识处理、评测能力应可独立迭代

---

## 8. 推荐实施顺序

不建议一开始就做大而全的平台，而是按以下顺序推进：

### 第 1 步：确定一个最小可验证场景

要求：

- 价值清晰
- 数据可获得
- 用户群明确
- 可在短周期内上线验证

### 第 2 步：输出项目评估报告

报告建议包含：

- 项目背景
- 用户与场景
- 数据条件
- 技术路线建议
- 风险分析
- MVP 范围
- 资源与周期估算

### 第 3 步：搭建单链路原型

先完成最小链路：

- 用户输入
- Prompt 编排
- 模型调用
- 结果展示

### 第 4 步：接入知识库或 RAG

补齐以下能力：

- 文档采集
- 文本切片
- 向量检索
- 结果重排
- 引用返回

### 第 5 步：接入工具调用

当业务需要系统执行能力时，再逐步加入：

- API 调用
- 数据库查询
- 文件处理
- 审批流或任务流

### 第 6 步：补齐治理能力

包括：

- 权限控制
- 日志与审计
- 监控与告警
- 成本追踪
- 评测体系

---

## 9. MVP 范围建议

一个实用的 MVP 版本通常只包含以下能力：

- 用户输入与输出界面
- Prompt 编排
- 模型调用
- 基础知识检索
- 简单日志记录
- 人工反馈收集

MVP 阶段不建议优先投入：

- 多 Agent 系统
- 复杂自治任务链
- 过度平台化抽象
- 过早私有化部署

---

## 10. 评测指标设计

不能只看模型是否“能回答”，还需要建立完整评测指标。

建议重点关注：

- 准确率：回答是否正确
- 可用率：是否真正帮助用户完成任务
- 稳定性：相同输入输出是否一致或可接受
- 时延：响应是否满足体验要求
- 成本：单次调用与整体使用成本
- 安全性：是否泄露敏感信息或越权
- 可维护性：Prompt、知识库和工具是否容易迭代

建议建立三类评测机制：

### 10.1 人工评测

由业务人员或测试人员对输出质量进行打分。

### 10.2 基准样本自动评测

建立一组固定问题集，对回答质量进行周期性回归检查。

### 10.3 线上真实反馈评测

采集真实用户反馈，作为后续优化依据。

## 11. LLM 辅助项目评估的提示词模板

可直接用于让模型生成项目评估报告：

```text
请你作为 AI 产品架构师和技术负责人，帮助我评估一个项目是否适合引入 LLM。

请从以下维度分析：
1. 业务目标与用户价值
2. 适合接入 LLM 的具体场景
3. 所需数据与知识来源
4. 技术实现路径（Prompt / RAG / Agent / Workflow）
5. 风险与限制
6. MVP 范围建议
7. 里程碑拆分
8. 推荐技术栈

输出格式要求：
- 先给结论
- 再给评估表
- 最后给实施建议
```

可直接用于让模型生成框架设计文档：

```text
1. 总体架构图说明

3. 数据流转路径

```

## 13. 项目启动时的交付物建议

建议在项目初期至少产出以下文档或结果：

- 项目评估报告
- 场景优先级清单
- 数据源清单
- 风险评估表
- MVP 范围说明
- 应用架构设计图
- 实施计划与里程碑
- 评测指标清单

---

## 14. 建议的落地策略

如果当前还处于方案阶段，建议优先完成：

1. 项目评估报告
2. 场景识别与优先级排序
3. MVP 方案设计

如果已经有明确场景，建议直接推进：

1. 单链路原型验证
2. RAG 或 Workflow 能力接入
3. 小范围用户试用
4. 基于反馈迭代框架设计

如果已经有现成代码仓库，建议进一步结合以下内容做专项设计：

- 当前模块结构
- 数据流和接口关系
- 可接入知识库的位置
- 可供 Tool Calling 调用的内部能力
- 安全、权限与审计要求

---

## 15. 总结

使用 LLM 对项目进行评估并构建应用框架，不建议从技术框架本身出发，而应该从业务目标、场景价值、数据条件、风险控制和 MVP 路线出发，逐步完成从验证到落地的全过程。

推荐顺序是：

1. 先评估是否值得做
2. 再判断适合做哪类 LLM 应用
3. 然后设计最小可行框架
4. 最后通过评测和反馈持续优化

这套方法既适合新项目规划，也适合在已有项目基础上逐步引入 LLM 能力。
